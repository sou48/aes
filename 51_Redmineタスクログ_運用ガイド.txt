　　　タスクログ評価方式　Redmine運用ガイド
　　　　　　　　　　　　タイガーラック株式会社 査定評価制度
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　本書は「07_タスクログ評価方式_設計書」の運用手順を、
　Redmineを使って実施するための具体的な設定・記録・集計方法を定めるものです。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　１．Redmineのセットアップ（管理者が初期設定）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ 1-1　カスタムフィールドの追加

　管理 → カスタムフィールド → チケット用 で以下を追加する。

　┌────────────────┬──────────┬────────────────────────────────────────┐
　│ フィールド名      │ 形式     │ 設定値（選択肢）                        │
　├────────────────┼──────────┼────────────────────────────────────────┤
　│ タスク種別        │ リスト   │ 制作 / 企画 / 改善 / 運用 / 調査・分析 / 支援・連携 │
　│ 難易度            │ リスト   │ XL / L / M / S                         │
　│ 主な成果・アウトプット│ 長文テキスト│ （自由記述・URL・数値を記入）      │
　└────────────────┴──────────┴────────────────────────────────────────┘

　　・「タスク種別」「難易度」は必須フィールドに設定する
　　・上記3フィールドをタスクログ対象プロジェクトのトラッカーに紐付ける

　■ 1-2　トラッカーの設定

　　・既存トラッカー（タスク・作業など）を利用するか、
　　　「査定タスク」という専用トラッカーを作成する
　　・カスタムフィールド3つを当該トラッカーに紐付ける

　■ 1-3　プロジェクト構成（推奨）

　　方式①　部署別プロジェクト（推奨）
　　　　例：「クリエイティブ制作課」「マーケティング企画部」などプロジェクトを部署単位で作成
　　　　→ 部長が自部署チケットのみ参照・管理できる

　　方式②　全社1プロジェクト
　　　　→ 横断的に見やすいが、評価者の権限管理が複雑になる

　■ 1-4　ステータスの設定

　　以下のステータスを用意する（既存ステータスに合わせて読み替え可）：

　　　・進行中（=作業中）
　　　・完了
　　　・継続中（次四半期へ持ち越し）
　　　・中断

　２．社員によるチケット記録ルール
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ 2-1　チケットの起票タイミング

　　・タスクに着手したタイミングで起票する（事後にまとめて書かない）
　　・週1回以上、進捗・成果を更新する
　　・完了時にステータスを「完了」に変更し、成果・アウトプット欄を最終更新する

　■ 2-2　チケットの記入ルール

　┌────────────────┬────────────────────────────────────────────────┐
　│ 項目             │ 記入内容・ルール                                   │
　├────────────────┼────────────────────────────────────────────────┤
　│ 題名（件名）     │ 「何を」「どのように」行ったかが分かる名称         │
　│                  │ 例：「○○社向けLPのデザイン・コーディング」        │
　│                  │ NG：「作業」「対応」などの曖昧な名称               │
　├────────────────┼────────────────────────────────────────────────┤
　│ 説明             │ タスクの背景・目的・対象を記載する                 │
　│                  │ 例：「新商品ページのリニューアルに伴い…」          │
　├────────────────┼────────────────────────────────────────────────┤
　│ タスク種別       │ 6分類から1つ選択（必須）                          │
　├────────────────┼────────────────────────────────────────────────┤
　│ 難易度           │ XL/L/M/S から1つ選択（必須）                      │
　│                  │ ※「自分にとっての難しさ」ではなく客観的複雑さで判断│
　│                  │ ※ 迷ったら1段階下（控えめ）に申告し面談で議論する │
　├────────────────┼────────────────────────────────────────────────┤
　│ 主な成果・        │ 完成物・URL・数値・変化を具体的に記入する          │
　│ アウトプット      │ 例：「https://～ 公開済み」「改善後の工数△30%削減」│
　├────────────────┼────────────────────────────────────────────────┤
　│ 担当者           │ 自分を設定（必須）                                │
　├────────────────┼────────────────────────────────────────────────┤
　│ 開始日 / 期日    │ 実際の着手日・完了日を入力                        │
　└────────────────┴────────────────────────────────────────────────┘

　■ 2-3　記録の粒度

　　・「打ち合わせ」「メール返信」などの細かい行動は1チケットにしない
　　・意味のあるアウトプット単位でまとめて1チケットとする
　　　例：「○○のLP制作」1チケット ← 打ち合わせ・構成・デザイン・コーディングをまとめる
　　・複数月にわたる大型タスクは1チケットで管理し、ステータスを「継続中」にする

　■ 2-4　作業記録（ジャーナル）の活用

　　・週次でチケットのジャーナル（コメント欄）に進捗を追記する
　　　例：「2026/04/10　構成案作成完了。来週デザイン着手予定」
　　　例：「2026/04/17　デザイン初稿完成。社内レビュー中」
　　・評価者が経緯を追えるため、ジャーナルの記入はログ品質向上に直結する

　３．四半期末の集計・エクスポート手順
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ STEP 1　チケット一覧のフィルタリング（本人 or 部長）

　　Redmine の「チケット一覧」画面で以下のフィルターを設定する：

　　　フィルター①：担当者 ＝ [評価対象者]
　　　フィルター②：開始日 ≧ [四半期開始日]
　　　フィルター③：開始日 ≦ [四半期終了日]
　　　（または 更新日 の範囲で代用可）

　　表示列に以下を追加する：
　　　・題名、タスク種別、難易度、ステータス、開始日、期日、主な成果・アウトプット

　■ STEP 2　CSVエクスポート

　　フィルタリング済みの一覧画面下部の「CSV」ボタンでエクスポートする。
　　エクスポートしたCSVを評価資料として保管する。

　■ STEP 3　集計（本人が四半期末に実施）

　　エクスポートしたCSVをExcel/スプレッドシートで開き、以下を集計する：

　　　・難易度別の完了チケット数（XL / L / M / S）
　　　・種別別の完了チケット数（制作 / 企画 / 改善 / 運用 / 調査・分析 / 支援・連携）
　　　・XL＋L比率（完了チケット中に占める割合）
　　　・支援・連携チケット数

　　集計結果と振り返りコメントをRedmineのWiki、または別途共有ドキュメントに記載する。
　　（※ Redmine WikiをサマリーページとしてURLで評価者に共有するのが理想）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　４．部長による評価手順（Redmine活用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ STEP 1　チケットログの確認

　　・担当者フィルターで対象者のチケット一覧を表示
　　・難易度・種別・ステータス・成果を横断的に確認する
　　・疑義のあるチケット（難易度の過大申告など）はジャーナルで
　　　本人にコメントを求めるか、面談で確認する

　■ STEP 2　難易度の修正権限

　　・部長はチケットの「難易度」フィールドを修正できる（最終権限）
　　・修正した場合はジャーナルに理由を記録すること
　　　例：「スコープが限定的なためLからMに修正」

　■ STEP 3　S/A/B/C判定

　　設計書（07_タスクログ評価方式_設計書）の判定マトリクスに照らし、
　　以下3軸それぞれに S/A/B/C を付ける：

　　　① 役割遂行（量・種別バランス）
　　　② 成果創出（質・難易度）
　　　③ 組織貢献（支援・連携）

　■ STEP 4　評価フィードバック通知書への転記

　　判定結果を「05_評価フィードバック通知書」に記入する。
　　Redmineの対象チケット一覧URLを通知書の備考欄に貼ると根拠が明確になる。

　５．Redmine活用の参考設定（推奨）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ クエリの保存（マイページ活用）

　　よく使うフィルター条件（自分の当該四半期チケット一覧）を
　　「カスタムクエリ」として保存しておくと、毎回設定する手間が省ける。
　　例：「2026年Q2_自分のタスク」という名前で保存

　■ ガントチャート・カレンダーとの併用

　　開始日・期日を入力することでガントチャートに自動反映され、
　　タスクの重なり・進捗の遅れを視覚的に把握できる。

　■ Wikiページで四半期サマリーを作成（任意）

　　各プロジェクト内のWikiに「2026年Q2_[氏名]_サマリー」ページを作成し、
　　集計数値・振り返り・自己評価を記載する。
　　部長は各自のWikiページを確認して評価コメントを記入できる。

　■ バージョン機能を四半期管理に活用（上級）

　　Redmineの「バージョン」機能を四半期名（例：2026年Q2）として作成し、
　　チケットに紐付けることで、四半期ごとのタスク絞り込みがより簡単になる。

　６．チケット記録の良い例・悪い例
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ 良い例

　　題名：　タイガーラック公式サイト トップページリニューアル
　　種別：　制作
　　難易度：L
　　成果：　デザイン・コーディング・CMS実装完了。表示速度スコア +15改善。
　　　　　　公開URL：https://www.tiger-rack.co.jp/
　　ジャーナル：
　　　　2026/04/05　ワイヤーフレーム確定
　　　　2026/04/12　デザイン初稿→部長承認
　　　　2026/04/20　コーディング完了・テスト中
　　　　2026/04/25　本番公開完了

　　▷ 何を・どのくらいの規模で・どんな成果が出たか、経緯も追える

　■ 悪い例

　　題名：　ウェブ対応
　　種別：　（未記入）
　　難易度：XL
　　成果：　完了

　　▷ 内容が不明・難易度の根拠がない・成果が確認できない
　　▷ このチケットは評価の根拠として使えない

　７．年次評価での活用
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　　・年間評価時は Q2〜Q4（または翌Q1）の全期間でフィルターして一覧化する
　　・難易度の成長推移（上期はM中心→下期はL増加など）も加点要素として活用する
　　・Redmineのチケット履歴が年次評価の一次証拠資料となる
