　　　タスクログ評価方式（Task Log Method）　設計書
　　　　　　　　　　　　タイガーラック株式会社 査定評価制度
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　１．方式の概要とコンセプト
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　数値目標（KPI）だけでは仕事の実態が見えにくい職種において、
　「四半期中に実際に行ったタスクの記録（ログ）」を評価根拠とする方式。

　■ 2つの評価方式の対比

　┌──────────────┬───────────────────────────┬───────────────────────────┐
　│                  │ KPI目標方式（営業部）        │ タスクログ方式（非数値部署）│
　├──────────────┼───────────────────────────┼───────────────────────────┤
　│ 評価の起点       │ 期初に設定した目標           │ 期中に積み上げたタスク記録  │
　│ 記録タイミング   │ 四半期初（目標設定）         │ 毎週・随時（継続的に記録）  │
　│ 評価の主軸       │ 目標対比の達成率             │ 量・質・貢献の総合          │
　│ 向いている仕事   │ 数値で測れる反復業務         │ プロジェクト型・創作・企画  │
　│ S/A/B/C判定      │ 達成率と基準の照合           │ ログ内容と判定マトリクスの照合│
　└──────────────┴───────────────────────────┴───────────────────────────┘

　■ 対象部署

　　・クリエイティブ事業部（制作課・開発課）
　　・マーケティング企画部
　　・事業推進部
　　※ 総務部は業務特性により部分的に適用（KPI方式と併用可）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　２．ログ記録項目の定義
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　タスクログ方式では、チケットごとに「何を行い、どのような成果が出たか」を
　明確に残すことを重視する。難易度や種別はRedmine上で必須登録しない。

　■ 記録する基本項目

　┌────────────┬──────────────────────────────────────────────────┐
　│ 項目             │ 内容                                               │
　├────────────┼──────────────────────────────────────────────────┤
　│ 題名             │ タスクの内容がひと目で分かる名称                   │
　│ 説明             │ 背景、目的、対象範囲、必要に応じて進め方を記載     │
　│ 主な成果・        │ 完成物、公開URL、数値変化、完了内容、進捗内容を記載│
　│ アウトプット      │                                                    │
　│ ステータス       │ 進行中 / 完了 / 継続中 / 中断                      │
　│ 担当者           │ 評価対象者本人                                      │
　│ 開始日 / 期日    │ 実際の着手日・完了予定日                            │
　└────────────┴──────────────────────────────────────────────────┘

　■ 補足観点

　　・他部署支援、後輩支援、改善提案、ナレッジ共有などは
　　　説明欄または成果欄に分かるように記載する
　　・評価者はチケット属性ではなく、成果物の内容、完了度、
　　　周囲への波及効果をもとに評価する


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　３．運用手順
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　STEP 1　タスクログの記録（本人・随時更新）

　　・Redmine にチケットを起票し、タスクごとに記録する（1タスク1チケット）
　　・細かく分けすぎず、意味のあるアウトプット単位でまとめる
　　・ステータスの更新はリアルタイムに行う
　　・記録が週1回以下の場合は評価の信頼性が落ちるため、こまめに更新すること
　　・Git管理が必要なタスクはチケット発行を必須とする
　　・Webサイトへの変更など「いつ・誰が・何をしたか」を後から振り返る必要がある作業についてはチケット発行を必須とする
　　・チケットは何らかのプロジェクト内に作成する
　　※ 詳細な記録方法は「51_Redmineタスクログ_運用ガイド.txt」を参照

　STEP 2　部長レビュー（評価者）

　　・Redmine で担当者フィルターをかけ、対象者の全チケットを確認する
　　・ログの量・質の妥当性を確認する
　　・判定マトリクス（第4章）に照らして評価軸ごとにスコアをつける
　　・「評価者コメント」欄に根拠を記入し、面談で本人にフィードバックする
　　※ 評価手順の詳細は「51_Redmineタスクログ_運用ガイド.txt」第4章を参照

　STEP 3　3軸評価への変換（部長）

　　タスクログの評価結果を、既存の3軸評価（役割遂行・成果創出・組織貢献）に変換する。

　　┌──────────────┬─────────────────────────────────────────────────┐
　　│ 3軸評価          │ タスクログ評価の対応                                 │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ① 役割遂行       │ タスク量（役職に期待される量を満たしているか）        │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ② 成果創出       │ タスク質（アウトプットの完成度・インパクト）          │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ③ 組織貢献       │ 他部署支援・ナレッジ共有・後輩支援・業務改善への     │
　　│                  │ 貢献度                                               │
　　└──────────────┴─────────────────────────────────────────────────┘

　　変換後のスコアを「40_統合評価シート_テンプレート.txt」または
　　「41_フィードバック通知書_テンプレート.txt」の評価項目欄に記入する。

　４．S/A/B/C 判定マトリクス
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ ① 役割遂行（量・進行管理）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ 期待タスク量の120%以上を完了。進捗更新も適時に行われ、          │
　│      │ 継続中タスクも含めて進行管理が非常に明確                        │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ 期待タスク量を概ね達成（100%前後）。進捗・完了記録も適切        │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ 期待量の80〜99%。または進捗更新や完了記録に抜け漏れがある      │
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ 期待量の80%未満。またはログが著しく少なく実態が不明            │
　└──────┴────────────────────────────────────────────────────────────┘

　■ ② 成果創出（質・インパクト）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ 高品質な成果を安定して出し、想定外のプラス成果                 │
　│      │ （売上寄与、品質向上、工数削減、社内外への波及効果など）がある │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ 期待される成果物を概ね高品質で完了している                      │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ 完了はしているが品質や成果の明確さに不足がある                  │
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ 未完了や品質不足が複数あり、成果として評価しにくい              │
　└──────┴────────────────────────────────────────────────────────────┘

　■ ③ 組織貢献（支援・共有・改善）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ 他部署や後輩への支援、共有、改善提案が複数あり、具体的な成果が │
　│      │ ある（改善定着・後輩の成長実績・社内ナレッジ蓄積など）         │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ 支援、共有、改善への関与が確認でき、確実に対応している          │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ 求められれば対応するが、自発的な支援や改善発信は少ない          │
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ 支援・共有・改善への関与がほぼなく、自分の業務のみ完結している │
　└──────┴────────────────────────────────────────────────────────────┘

　■ 期待タスク量の目安（役職・グレード別）

　┌──────────┬────────────────────────────────────────────────────┐
　│ 役職       │ 四半期の期待タスク構成（目安）                           │
　├──────────┼────────────────────────────────────────────────────┤
　│ 一般（G1） │ 日常担当タスクを安定して完了し、成果記録を継続できること│
　│ 一般（G2） │ 標準担当業務を自走し、品質の安定した成果を出すこと      │
　│ 一般（G3） │ 主要担当を任され、継続タスクも含めて安定推進すること    │
　│ 副主任（G4）│ 自身の担当に加え、後輩支援や改善提案が見られること      │
　│ 主任（G5） │ チーム内の主要案件を主導し、成果の再現性を作ること      │
　│ 係長〜（G6+）│ 部署横断の推進、改善定着、人材育成への貢献を出すこと  │
　└──────────┴────────────────────────────────────────────────────┘

　５．ログ運用の注意事項
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　● 記録の粒度
　　・「打ち合わせ」「メール返信」などの細かい行動はログしない
　　・「○○のLP制作」「○○の改善施策立案・実行」など、
　　　意味のあるアウトプット単位でまとめて記録する

　● ログを埋めてから評価するのではなく、日々の仕事をそのまま記録する
　　→ 四半期末に慌てて記入したログは評価の信頼性が下がる

　● ステータス「継続中」のタスクの扱い
　　・四半期をまたぐ大型タスクは「継続中」のまま記録する
　　・進捗状況（どこまで完了したか）を「主な成果・アウトプット」欄に記載する
　　・完了は次四半期に記録し、評価も次四半期分に計上する

　● 年次総合評価への反映
　　・年間の全タスクログを通して評価する場合は、
　　　4四半期分のログを統合して判断する（成果の安定性や支援範囲の広がりも加点要素）
