　　　タスクログ評価方式（Task Log Method）　設計書
　　　　　　　　　　　　タイガーラック株式会社 査定評価制度
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　１．方式の概要とコンセプト
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　数値目標（KPI）だけでは仕事の実態が見えにくい職種において、
　「四半期中に実際に行ったタスクの記録（ログ）」を評価根拠とする方式。

　■ 2つの評価方式の対比

　┌──────────────┬───────────────────────────┬───────────────────────────┐
　│                  │ KPI目標方式（営業部）        │ タスクログ方式（非数値部署）│
　├──────────────┼───────────────────────────┼───────────────────────────┤
　│ 評価の起点       │ 期初に設定した目標           │ 期中に積み上げたタスク記録  │
　│ 記録タイミング   │ 四半期初（目標設定）         │ 毎週・随時（継続的に記録）  │
　│ 評価の主軸       │ 目標対比の達成率             │ 量・質・難易度・貢献の総合  │
　│ 向いている仕事   │ 数値で測れる反復業務         │ プロジェクト型・創作・企画  │
　│ S/A/B/C判定      │ 達成率と基準の照合           │ ログ内容と判定マトリクスの照合│
　└──────────────┴───────────────────────────┴───────────────────────────┘

　■ 対象部署

　　・クリエイティブ事業部（制作課・開発課）
　　・マーケティング企画部
　　・事業推進部
　　※ 総務部は業務特性により部分的に適用（KPI方式と併用可）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　２．タスクの分類定義
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ タスク種別（6分類）

　┌────────┬──────────────────────────────────────────────────────────┐
　│ 種別     │ 内容・例                                                       │
　├────────┼──────────────────────────────────────────────────────────┤
　│ 制作     │ Webサイト制作、バナー、動画、ツール開発、システム実装          │
　│ 企画     │ 施策立案、提案書作成、キャンペーン設計、改善計画               │
　│ 改善     │ 既存業務・仕組みの改善実施、プロセス変更、効率化               │
　│ 運用     │ 定期更新、メンテナンス、監視、日次・週次の定型対応             │
　│ 調査・分析│ 競合調査、効果測定、データ分析、レポート作成                  │
　│ 支援・連携│ 他部署サポート、勉強会・共有、後輩指導、社内問合せ対応        │
　└────────┴──────────────────────────────────────────────────────────┘

　■ タスク難易度（4段階）

　┌──────┬────────┬──────────────────────────────────────────────────┐
　│ 難易度 │ 目安期間 │ 判断基準                                               │
　├──────┼────────┼──────────────────────────────────────────────────┤
　│  XL   │ 1ヶ月以上 │ 複数人・複数部署が関わる大規模プロジェクト、           │
　│       │          │ 戦略的意思決定を伴う企画、新規サービス・機能の構築     │
　├──────┼────────┼──────────────────────────────────────────────────┤
　│  L    │ 1〜4週間 │ 独立した企画・制作・開発1案件。スキルと判断を要する    │
　│       │          │ 標準的な業務の主要タスク                               │
　├──────┼────────┼──────────────────────────────────────────────────┤
　│  M    │ 数日〜1週 │ 定型化されていない対応、改修・部分制作、小規模調査     │
　├──────┼────────┼──────────────────────────────────────────────────┤
　│  S    │ 1日以内  │ 定型業務・修正・問合せ対応・定期更新などの軽微な作業   │
　└──────┴────────┴──────────────────────────────────────────────────┘

　※ 難易度は「かかった時間」ではなく「必要な判断・スキルの複雑さ」で判定する。
　※ 評価者（部長）が最終的に難易度を確認・修正できる。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
　３．運用手順
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　STEP 1　タスクログの記録（本人・随時更新）

　　・Redmine にチケットを起票し、タスクごとに記録する（1タスク1チケット）
　　・細かく分けすぎず、意味のあるアウトプット単位でまとめる
　　・ステータスの更新はリアルタイムに行う
　　・記録が週1回以下の場合は評価の信頼性が落ちるため、こまめに更新すること
　　・Git管理が必要なタスクはチケット発行を必須とする
　　・Webサイトへの変更など「いつ・誰が・何をしたか」を後から振り返る必要がある作業についてはチケット発行を必須とする
　　・チケットは何らかのプロジェクト内に作成する
　　※ 詳細な記録方法は「51_Redmineタスクログ_運用ガイド.txt」を参照

　STEP 2　部長レビュー（評価者）

　　・Redmine で担当者フィルターをかけ、対象者の全チケットを確認する
　　・ログの量・質の妥当性を確認する
　　・判定マトリクス（第4章）に照らして評価軸ごとにスコアをつける
　　・「評価者コメント」欄に根拠を記入し、面談で本人にフィードバックする
　　※ 評価手順の詳細は「51_Redmineタスクログ_運用ガイド.txt」第4章を参照

　STEP 3　3軸評価への変換（部長）

　　タスクログの評価結果を、既存の3軸評価（役割遂行・成果創出・組織貢献）に変換する。

　　┌──────────────┬─────────────────────────────────────────────────┐
　　│ 3軸評価          │ タスクログ評価の対応                                 │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ① 役割遂行       │ タスク量（役職に期待される量を満たしているか）        │
　　│                  │ ＋ 種別バランス（運用偏りすぎていないか）             │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ② 成果創出       │ タスク質（アウトプットの完成度・インパクト）          │
　　│                  │ ＋ 難易度（XL/Lタスクへの挑戦比率）                  │
　　├──────────────┼─────────────────────────────────────────────────┤
　　│ ③ 組織貢献       │ 支援・連携タスクの量と質                             │
　　│                  │ ＋ 他者の成長・業務改善への貢献度                    │
　　└──────────────┴─────────────────────────────────────────────────┘

　　変換後のスコアを通知書（05_）の評価項目欄に記入する。

　４．S/A/B/C 判定マトリクス
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　■ ① 役割遂行（量・種別）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ 期待タスク量の120%以上完了。かつ制作・企画・改善が中心で       │
　│      │ 運用タスクが全体の30%以下に抑えられている                      │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ 期待タスク量を概ね達成（100%前後）。種別バランスも適切         │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ 期待量の80〜99%。または運用タスク偏重（50%超）で実態に伴わない│
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ 期待量の80%未満。またはログが著しく少なく実態が不明            │
　└──────┴────────────────────────────────────────────────────────────┘

　■ ② 成果創出（質・難易度）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ XLまたはL難易度タスクが全体の30%以上。かつ高品質で完了。      │
　│      │ 想定外のプラス成果（社内外への波及効果）が1件以上ある          │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ Lタスクが全体の20%前後含まれ、概ね高品質で完了                 │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ Lタスクが10%未満でS/Mが中心。または完了タスクに品質不足あり   │
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ ほぼすべてS/M難易度で挑戦なし。または未完了・品質不足が複数   │
　└──────┴────────────────────────────────────────────────────────────┘

　■ ③ 組織貢献（支援・連携）の判定

　┌──────┬────────────────────────────────────────────────────────────┐
　│  S   │ 他部署や後輩への支援・共有タスクが複数あり、具体的な成果が     │
　│      │ ある（改善定着・後輩の成長実績・社内ナレッジ蓄積など）         │
　├──────┼────────────────────────────────────────────────────────────┤
　│  A   │ 支援・連携タスクが1件以上あり、確実に対応している              │
　├──────┼────────────────────────────────────────────────────────────┤
　│  B   │ 求められれば対応するが自発的な支援は少ない                     │
　├──────┼────────────────────────────────────────────────────────────┤
　│  C   │ 支援・連携タスクがほぼなく、自分の業務のみ完結している         │
　└──────┴────────────────────────────────────────────────────────────┘

　■ 期待タスク量の目安（役職・グレード別）

　┌──────────┬────────────────────────────────────────────────────┐
　│ 役職       │ 四半期の期待タスク構成（目安）                           │
　├──────────┼────────────────────────────────────────────────────┤
　│ 一般（G1） │ S・Mタスク中心。Lタスク1〜2件あれば合格ライン           │
　│ 一般（G2） │ M・Lタスク中心。Lタスク3件以上がA評価ライン             │
　│ 一般（G3） │ Lタスク中心。XLへの参加・補佐実績があればS候補          │
　│ 副主任（G4）│ Lタスクをリードかつ後輩指導タスク1件以上               │
　│ 主任（G5） │ XL・Lタスクのオーナーとして推進。XL1件以上がA基準      │
　│ 係長〜（G6+）│ XLタスクを複数主導。部署横断の成果を出すこと          │
　└──────────┴────────────────────────────────────────────────────┘

　５．ログ運用の注意事項
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

　● 記録の粒度
　　・「打ち合わせ」「メール返信」などの細かい行動はログしない
　　・「○○のLP制作」「○○の改善施策立案・実行」など、
　　　意味のあるアウトプット単位でまとめて記録する

　● 難易度の自己申告ルール
　　・難易度は「自分にとっての難しさ」ではなく「客観的な業務複雑性」で判断する
　　・迷った場合は1段階下（控えめ）に申告し、面談で評価者と議論する

　● ログを埋めてから評価するのではなく、日々の仕事をそのまま記録する
　　→ 四半期末に慌てて記入したログは評価の信頼性が下がる

　● ステータス「継続中」のタスクの扱い
　　・四半期をまたぐ大型タスクは「継続中」のまま記録する
　　・進捗状況（どこまで完了したか）を「主な成果・アウトプット」欄に記載する
　　・完了は次四半期に記録し、評価も次四半期分に計上する

　● 年次総合評価への反映
　　・年間の全タスクログを通して評価する場合は、
　　　4四半期分のログを統合して判断する（難易度の成長推移も加点要素）
