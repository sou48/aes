## 2. 制度の基本設計方針

### 設計優先順位（重要）

1. **価値観・行動基準の統一**（最優先）
2. 公平な昇給決定
3. 人材育成
4. マネジメント責任の明確化
5. 業績向上

> 昇給の最大化・業績向上は目的の末尾。「正しく評価し、正しく処遇する」が本制度の核心。

### 評価の3軸（全社共通）

| 軸 | 内容 |
|---|---|
| ① 役割遂行 | 役職・職種に期待される業務を果たしているか |
| ② 成果創出 | 数値・成果物・改善など具体的なアウトプットを出しているか |
| ③ 組織貢献 | チーム・他部署・会社全体への貢献・情報共有・育成など |

軸ごとのウェイトは部署・役職によって異なる（`30_評価項目_部署別設計.txt` 参照）。
部署ごとの評価項目は、適用方式（KPI / タスクログ / 行動基準 / KPI＋タスクログ併用）に応じて
`30_評価項目_部署別設計.txt` で定義する。
同一部署でも担当業務が異なるため、実際の適用項目は当該部署の部長が四半期初に候補項目から選択する。

### 評価スケール

| 評価 | 意味 | スコア目安 |
|---|---|---|
| S | 期待を大きく上回る | 3.50点以上 |
| A | 期待通り | 3.00〜3.49点 |
| B | やや不足 | 2.00〜2.99点 |
| C | 不足・改善が必要 | 2.00点未満 |

中間評価なし（4段階固定）。

## 3. 等級グレード設計の扱い

現時点では、固定的な「等級グレード制度」はペンディングとする。

理由は、従業員の給与改定が以下の流れで決まるためである。

1. 決算確定後、役員が各部署へ昇給原資をブロック（昇給単位）で割り振る
2. 各部署責任者が四半期評価の結果をもとに、部署内でそのブロックを配分する
3. その配分結果によって翌年度の昇給額を決定する

このため、現段階では「役職に応じた固定グレードを先に置く」よりも、
「評価結果に応じて昇給ブロックを配分する」運用のほうが制度実態に合っている。

### 当面の整理

- 役職の呼称・責任範囲は維持する
- 四半期評価は昇給配分の根拠として使用する
- 等級グレード・号俸・給与テーブルの制度化は保留する

### 将来の再設計方針

査定評価制度が安定稼働した後、その時点の全従業員の実給与をもとに等級グレードを再設計する。

詳細は `10_等級定義_グレード一覧.txt` を参照。

## 4. 評価方式（3方式）

部署・役職に応じて以下3方式のいずれかを適用する。どの方式を使っても最終的には統合評価シート（`40_統合評価シート_テンプレート.txt`）に集約される。

### 方式の概要

| 方式 | 概要 | 向いている業務 |
|---|---|---|
| ① KPI方式 | 期初に数値目標を設定し、達成率で評価 | 売上・件数・率など定量指標で測れる業務 |
| ② タスクログ方式 | 四半期中のRedmineタスク記録を評価根拠とする | プロジェクト型・企画・制作など定性的な業務 |
| ③ 行動基準評価方式 | 行動基準チェックリストへの準拠と上司観察で評価 | 定型・反復・チーム完結型の現場オペレーション |

### 部署別マッピング（要約）

| 評価方式 | 対象 |
|---|---|
| ① KPI | 営業課（全員）、施工事業部管理職、物流事業部管理職、全部署の部長職 |
| ② タスクログ | クリエイティブ事業部制作課・開発課（全員）、事業推進部（全員） |
| ①② 併用 | マーケティング企画部（全員）、総務部管理職 |
| ③ 行動基準 | 受注販売課（全員）、施工事業部スタッフ（現場）、物流事業部スタッフ（倉庫）、総務部スタッフ（事務） |

> 詳細：`20_評価方式_部署別マッピング.txt`、`24_評価方式_部署別マッピング一覧.csv`

### 管理職昇格時の方式切替

行動基準評価（③）対象者が副主任（G4）以上に昇格した場合、翌四半期から以下に切り替える：

| 部署 | スタッフ時 | 管理職昇格後 |
|---|---|---|
| 施工事業部 | ③ 行動基準 | ① KPI |
| 物流事業部 | ③ 行動基準 | ① KPI |
| 総務部 | ③ 行動基準 | ①② KPI＋タスクログ |

## 5. 評価フロー（四半期ごと）

```
WEEK 1〜2   期初：KPI設定または目標確認（部長承認）
WEEK 3〜12  実務期間（必要に応じて中間確認）
WEEK 13     評価入力（部長が各社員を採点。部長自身は自己評価を提出）
WEEK 14     キャリブレーション（役員が分布確認・逸脱部署のみ個別レビュー）
WEEK 15     面談実施（部長→社員。統合評価シートの通知書パートを使用）
WEEK 16     通知書の保管（会社1部・本人1部）
```

## 6. キャリブレーション（評価補正）

> **方針変更済み**：従来の「役員＋全部長による横断会議」方式は廃止。
> 他部署業務を知らない部長同士が相互評価することが現実的に困難なため、以下の2段階方式に変更。

### STEP 1：評価分布の自己チェック（部長が実施）

各部長は評価確定前に自部署のS/A/B/C比率をガイドラインと照合する：

| 評価 | 推奨比率 |
|---|---|
| S | 10〜15% |
| A | 30〜40% |
| B | 40〜50% |
| C | 5〜10% |

> 対象者3名以下の部署はガイドライン比率を免除（役員レビューのみ）

### STEP 2：S/C評価の根拠文提出（部長→役員）

S評価者・C評価者全員について、具体的な行動・成果・事実を含む根拠文（3〜5行）を役員に提出する。

### STEP 3：役員による個別レビュー（逸脱部署のみ・1対1）

S比率超過・全員A以上・業績との乖離などのトリガーに該当した部署について、役員が当該部長と1対1で確認。他部署の部長は不関与。

## 7. 年次昇給フロー

```
12月末     ：年次総合評価確定（Q1〜Q4の加重平均）
翌年2月末  ：決算確定 → 昇給原資（総ステップ本数）算出
翌年3月    ：役員が部署別ステップ本数を配分
             部長が部署内のステップ本数を各社員に割り当て
翌年4月    ：新給与適用（4月支給分から反映）
```

### 昇給ステップ数の目安

| 評価 | ステップ本数 | 月額換算（+） |
|---|---|---|
| S | 6〜8本 | +9,000〜12,000円 |
| A | 4〜5本 | +6,000〜7,500円 |
| B | 2〜3本 | +3,000〜4,500円 |
| C | 0〜1本 | 0〜+1,500円 |

- 部署業績係数（好調×1.2 / 標準×1.0 / 不振×0.8）を乗じて配分本数が変わる
- 同評価ランク内での部長裁量調整は可（ランクを超えた逸脱は禁止）

### 初年度（2026年度）の特例

- Q1（1〜3月）は制度導入準備期間のためスキップ
- Q2・Q3・Q4（3四半期）の均等加重で年次総合評価を算出
- 2027年4月が初回の昇給反映月

## 8. 統合評価シートの構造

すべての評価方式・部署で共通の1枚の評価シート（`40_統合評価シート_テンプレート.txt`）を使用する。

```
Section 1  方式別評価根拠
  ├─ 1-A  KPI方式用（目標・実績・達成率）
  ├─ 1-B  タスクログ方式用（タスク実績・成果・件数）
  └─ 1-C  行動基準方式用（チェックリスト結果）
     ↓ 方式に応じていずれか1つを記入

Section 2  3軸スコア集計（全方式共通）
  役割遂行・成果創出・組織貢献のウェイト付き加重平均 → 総合評価（S/A/B/C）

Section 3  フィードバック通知書（本人へ渡す）
  評価結果・強み・改善点・次四半期アクション

Section 4  評価者署名・確認欄

Section 5  年間累積シート
  Q1〜Q4の評価を転記 → 年次総合評価 → 昇給ステップ数
```

## 9. 部長評価の特別ルール

全部署の部長（G9）は「KPI方式＋部署独自評価3項目」で評価される。評価者は役員（部長自身は自己評価のみ提出）。

### 部署独自3項目の構成

各部長について、役割遂行・成果創出・組織貢献それぞれ1項目ずつ、部署の役割・ミッションに基づいた独自項目を設定（`31_評価項目_部長独自項目.txt` 参照）。
共通項目としては、与えられた部門目標の達成だけでなく、
新しいサービス・商品・仕組みの創出や、メンバーから挑戦案を引き出すマネジメントも重視する。
部長評価のウェイトは、他部署と同様の百分率表記で
役割遂行 45% ／ 成果創出 30% ／ 組織貢献 25% を基本とする。

### 部長評価が減点されるケース

- 部署全員A以上かつ業績が標準以下（評価インフレ）
- 根拠文不備が四半期で2件以上
- 役員レビューによる評価修正が四半期で3件以上
- 部門目標未達なのにS評価者が複数いる

## 10. タスクログ方式（Redmine）の運用概要

タスクログ方式は、本人が四半期中にRedmineを随時更新し、その記録を部長がレビューして評価する。
四半期末のCSVエクスポートやサマリー作成は必須作業ではなく、レビューや説明に必要な場合のみ補助的に行う。
3軸への変換は、役割遂行を「役職に期待されるタスク量」、成果創出を「成果物の完成度・インパクト」、
組織貢献を「支援・共有に加え、他者成長や業務改善への貢献」として判定する。

### Redmine記録の基本項目

| 項目 | 内容 |
|---|---|
| 題名 | タスク内容が分かる名称 |
| 説明 | 背景・目的・対象範囲 |
| 主な成果・アウトプット | 完成物、URL、数値変化、進捗内容 |
| ステータス | 進行中 / 完了 / 継続中 / 中断 |
| 担当者 | 評価対象者本人 |
| 開始日 / 期日 | 着手日・完了予定日 |

難易度・種別はRedmine上で登録しない。評価はチケット属性ではなく、
ログ件数、成果物の質、進捗更新の適切さ、支援・改善の記録内容をもとに行う。

> 詳細：`22_タスクログ評価方式_設計.txt`、`51_Redmineタスクログ_運用ガイド.txt`

## 11. 行動基準評価方式の適用部署

受注販売課、施工・物流・総務の定型・反復業務に適用。各部署の行動基準項目と上司観察で評価。

### 強制C条件（どの評価点でも総合評価がCになる）

- 安全事故を起こした（または隠蔽した）
- 重大ミスを繰り返した（同じミスが四半期内3回以上）
- 上司・同僚への著しく不誠実な対応

> 詳細：`23_行動基準評価方式_設計.txt`

## 12. 設計上の重要な意思決定ログ

後続エージェントが判断に迷った際の参考として、主要な設計判断の経緯を記録する。

| 判断事項 | 採用した方針 | 理由 |
|---|---|---|
| 評価方式 | 3方式（KPI / タスクログ / 行動基準）を部署・役職別に適用 | 物流・施工スタッフにはKPIもタスクログも合わない「第3のパターン」が存在したため |
| キャリブレーション | 全部長会議方式 → 廃止。評価分布チェック＋役員個別レビューに変更 | 他部署業務を知らない部長同士の相互評価は形骸化するため |
| 昇給タイミング | 4月（1月ではない） | 12月決算→2月末決算確定→3月割り振り→4月反映というフローのため |
| 統合評価シート | 方式別に分かれず「Section1のみ方式別・Section2以降は共通」の1枚 | どの方式でも共通の評価シートで完結させるため |
| マーケ・総務管理職 | ①②KPI＋タスクログの併用方式 | 定量KPIと定性タスクが混在する業務性質のため |
| 等級グレード | 現時点では制度化せずペンディング | 給与決定が「昇給ブロックの配分」で行われるため、固定グレード設計と整合しないため |
| 部長評価の重点 | ミッション遂行に加えて「創出」と「挑戦喚起」を評価に含める | 責任者には既存業務の遂行だけでなく、新しい成長機会を生み出す役割を期待するため |
