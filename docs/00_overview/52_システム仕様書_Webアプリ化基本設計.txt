# 52_システム仕様書_Webアプリ化基本設計

タイガーラック株式会社 査定評価システムをWebアプリ化するための基本仕様書。
本書は、制度設計ドキュメント群をシステム要件へ落とし込むための基準文書とする。

---

## 1. 目的

- 四半期評価、年次評価、昇給配分、給与正本反映までを一貫してWeb上で管理する
- 紙・Excel中心運用を廃止し、履歴追跡・権限制御・監査性を確保する
- 評価制度の運用ルールをシステム側へ固定し、部署間の運用ブレを抑える

---

## 2. システム化対象

- 四半期評価シートの作成・編集・公開
- KPI入力、KPI実績入力、上長ロック
- Redmine連携によるタスクログ参照
- 行動基準評価項目の選択と採点
- 3軸評点、総合評価点、年次総合評価点の算出
- 年次総括コメント、年次通知書生成
- 部署別昇給ブロック配分、社員別昇給本数入力
- 給与正本への仮反映・確定反映
- 通知メール、監査ログ、マスタ管理

---

## 3. 権限設計

### 3-1. 評価権限ロール

- 役員
- 部長
- 管理職
- 一般

### 3-2. 実際の役職

- 役員
- 部長
- 課長
- 係長
- 主任
- 副主任
- 一般

### 3-3. 実際の役職と評価権限ロールの初期紐付け

- 役員 → 役員
- 部長 → 部長
- 課長 → 管理職
- 係長 / 主任 / 副主任 / 一般 → 一般

評価権限ロールは実際の役職から自動初期設定するが、システム管理者が上書き可能とする。

### 3-4. 操作範囲

- 役員: 全社員の評価内容、昇給配分、面談メモ、監査ログを閲覧可能。配分確定可能
- 部長: 自部署の管理職・スタッフを操作閲覧可能
- 管理職: 自部署スタッフを操作閲覧可能
- 一般スタッフ: 自分のKPI入力、自分の公開済み通知書閲覧、評価根拠履歴閲覧のみ
- 総務部: 給与正本のうち `確定昇給本数 / 確定昇給額 / 反映年月` のみ閲覧・仮入力可能
- システム管理者: マスタとシステム設定のみ変更可能。評価・昇給の判断操作は不可

### 3-5. ロール付与

- 1ユーザーに複数ロール付与可能
- 役員ロール保有者は評価対象外
- 役員であっても、システム管理者ロールがなければ人事マスタ変更は不可

---

## 4. ユーザー・組織・マスタ

### 4-1. ユーザーマスタ

最低限持つ項目:

- 社員コード
- 氏名
- フリガナ
- 会社メールアドレス
- 在籍区分
- 本部
- 部
- 課
- 実際の役職
- 評価権限ロール
- 一次評価者

補足:

- 社員コードはシステム採番、変更不可、再利用不可
- 会社メールアドレスはログインIDとする
- 役員のみ一次評価者を未設定可とする

### 4-2. 在籍区分

- 在職中
- 休職中（産休育休 / 傷病 / その他）
- 退職予定
- 退職済

操作ルール:

- 休職中はログイン可、過去評価閲覧のみ可
- 退職予定は退職日まで通常操作可
- 退職済はログイン不可、過去データのみ保持

### 4-3. 組織マスタ

- 本部マスタ
- 部マスタ
- 課マスタ

管理ルール:

- 本部 → 部 → 課 の親子構造
- コードはシステム採番、変更不可
- 課が存在しない部署は課を空欄管理

### 4-4. 部署別マッピングマスタ

キー:

- 本部
- 部
- 課
- 評価権限ロール

保持項目:

- 既定評価方式

ルール:

- 課まで一致を優先し、なければ部まで一致を使用
- 同一キーの重複登録は不可
- 未設定時は評価シート作成不可
- 未設定時はシステム管理者へメール通知

### 4-5. マスタ共通ルール

- 編集はシステム管理者のみ
- 役員は閲覧のみ
- 変更履歴は `変更日 / 変更前 / 変更後 / 変更者`
- 変更履歴閲覧は役員とシステム管理者のみ

---

## 5. 認証とアカウント

- 認証方式は会社メールアドレス + パスワード
- 初回アカウントは従業員名簿をもとに事前作成
- 初回利用は再設定リンク経由で開始
- パスワード要件は8文字以上
- パスワード再設定は登録メールアドレス宛リンク送信
- 自動タイムアウトは設けない

---

## 6. 評価シートの基本ルール

- 同一社員・同一年度・同一四半期につき評価シートは1件のみ
- 四半期評価シートは評価者が作成
- 保存時点で確定扱いだが、印刷までは修正可
- 印刷後は役員のみ修正可
- 印刷後に役員修正した場合は再印刷要フラグを立てる

### 6-1. 管理番号

- 四半期評価シート: `年度-Q1〜Q4-部署コード-社員コード`
- 年次通知書: `年度-YR-部署コード-社員コード`
- 管理番号は一覧 / 詳細 / PDF / 監査ログに表示

### 6-2. 進捗ステータス

- 未作成
- 作成中
- 面談未完了
- 公開待ち
- 公開済

昇給配分は別に以下の状態を持つ:

- 下書き
- 役員確認待ち
- 差し戻し
- 確定

---

## 7. 評価方式

評価方式は固定4択とし、マスタ追加は不可:

- KPI方式
- タスクログ方式
- 行動基準評価方式
- KPI+タスクログ方式

### 7-1. 初期評価方式

- 最優先: 直近前回シートの評価方式
- なければ: 部署別マッピングマスタの既定評価方式

### 7-2. 評価方式変更

- 各四半期の最初の1ヶ月間のみ変更可
- 変更権限は一次評価者、部長、役員
- 変更時は既存の根拠データをクリア
- 本人と一次評価者へ通知メール送信

### 7-3. KPI+タスクログ方式

- 独立した1つの評価方式として扱う
- 3軸割当は固定
  - 成果創出 = KPI
  - 役割遂行 = タスクログ
  - 組織貢献 = タスクログ
- 本人はKPI入力のみ可能
- 本人向け評価根拠履歴ではKPI部分のみ表示

---

## 8. KPI方式

### 8-1. KPI設定

- 1期あたり3〜5件
- 3件未満でも保存は可だが未完了警告を表示
- 提出は3件以上必須
- 各KPIの必須項目
  - 名称
  - 目標内容
  - 完了条件
  - 重要順

### 8-2. KPI重み

- 被評価者が重要順を決める
- システムが件数と並び順に応じて固定ウェイトを自動設定
- 被評価者に重みは見せる

### 8-3. KPI入力と実績入力

- 目標も実績も被評価者本人が入力し、上長が確認・修正
- 期限後も上長未ロックなら本人編集可
- 上長ロック後は上長のみ直接編集可
- 各KPIに `達成 / 一部達成 / 未達` 状態を上長が付与

### 8-4. KPI履歴

- 本人は過去のKPI履歴を閲覧可
- 表示項目
  - 年度
  - 四半期
  - KPI名称
  - 目標内容
  - 完了条件
  - 実績
  - 達成状態
  - 根拠メモ
- 画面閲覧のみ、PDF出力なし
- スマホ閲覧可

---

## 9. タスクログ方式

- Redmine API連携を第一候補とする
- API不可時はCSVインポートで代替
- 対象は、当該四半期中に更新された本人担当チケットのみ
- 一覧は全件表示、更新日時の新しい順
- 私用や雑務も原則一覧に出し、評価者が評価対象か判断する
- 本人向けにはタスクログ履歴を表示しない

---

## 10. 行動基準評価方式

- 3軸それぞれ3〜5項目を選択
- 各項目は評価者が S / A / B / C を付与
- 各項目評点と、3軸最終評点を別欄で持つ
- 3軸最終評点は評価者が確定する
- 本人向け履歴では以下を表示
  - 年度
  - 四半期
  - 項目名
  - 項目評点
  - 3軸最終評点
- 本人にはコメントや内部メモは見せない
- スマホ閲覧は不要

---

## 11. 評点・点数・年次集計

### 11-1. 3軸評点

- 役割遂行
- 成果創出
- 組織貢献

全方式共通で、最終的な3軸評点は評価者が S / A / B / C で確定する。

### 11-2. 総合評価点

- S = 100
- A = 75
- B = 50
- C = 25

各軸のウェイトをかけて総合評価点を算出し、四捨五入で整数表示する。
総合ランクは出さず、点数のみ表示する。

### 11-3. 年次総合評価点

- Q1〜Q4を均等25%で集計
- 評価対象外四半期がある場合は、勤務日数按分で再計算
- 年次画面には以下を表示
  - Q1〜Q4点数
  - 各四半期の勤務日数按分比率
  - 年次総合評価点

### 11-4. コメント

- 3軸ごとの評点理由コメントは必須
- 総合コメントは必須
- 年次総括コメントは任意

---

## 12. 面談・本人公開

### 12-1. 四半期面談

- 面談実施日入力で完了
- 面談メモは任意
- 面談メモは評価者・役員のみ閲覧可

### 12-2. 年次面談

- 年次面談実施日入力で完了
- 年次面談メモは持たない

### 12-3. 公開ルール

- 四半期通知書と評価根拠履歴は四半期面談完了後に公開
- 年次通知書は年次面談完了後に公開
- 公開済みは取消可
- 四半期通知書の公開取消時は評価根拠履歴も連動して非公開
- 年次通知書の公開取消は四半期系には影響しない
- 本人画面では未公開行を `年度 / 四半期 / 未公開` で表示

---

## 13. 出力物

### 13-1. 四半期通知書PDF

表示項目:

- 管理番号
- 発行日
- 対象者名
- 所属（部 / 課）
- 役職
- 評価方式
- 一次評価者名

- 総合評価点
- 3軸評点
- 3軸評点理由
- 総合コメント
- 前回点数
- 前回3軸評点

補足:

- 総合は点数のみ表示
- 前回値がない場合は `前回値がないため比較なし`

### 13-2. 年次通知書PDF

表示項目:

- 年次通知書管理番号
- 発行日
- 対象者名
- 所属（部 / 課）
- 役職
- 一次評価者名
- 年次総合評価点
- Q1〜Q4総合評価点一覧
- 3軸推移グラフ
- 年次総括コメント

グラフ仕様:

- 3本線グラフ
- 縦軸は1〜4
- 評価がない四半期は欠損表示

昇給情報はシステム上保持するが、年次通知書PDFには現時点で表示しない。

### 13-3. PDF出力ルール

- PDFは都度生成のみ、ファイル保存はしない
- 版数・出力日時・出力者は履歴として保持
- 履歴閲覧は役員と評価者のみ

## 14. 昇給配分

### 14-1. 部署ブロック配分

- 年度ごとに役員が部署単位で総ブロック数を設定
- 部長が管理職・スタッフへ手入力配分
- 部長自身は役員が別画面で配分

### 14-2. 推奨昇給本数

- 部署内対象者の年次総合評価点比率で自動按分
- 小数は切り捨て
- 余りは高得点順で配布
- 同点時優先順位
  - 成果創出
  - 役割遂行
  - 組織貢献
  - 最後は部長手動判断

### 14-3. 配分フロー

- 保存時: 下書き
- 提出時: 役員確認待ち
- 差し戻し可、理由コメント必須
- 差し戻し後は部長が再編集・再提出
- 確定後はロック

---

## 15. 給与正本

- 対象はまず `査定による昇給額・昇給履歴`
- 将来は基本給・手当・確定支給額まで拡張可能な前提で設計

### 15-1. 権限

- 総務部: 仮入力、閲覧
- 役員: 確定、閲覧

### 15-2. 状態

- 仮
- 確定

### 15-3. 修正ルール

- 確定後修正は上書きではなく、新しい仮レコードを追加して再確定
- 変更理由必須

### 15-4. 履歴項目

- 変更日
- 変更前金額
- 変更後金額
- 変更理由
- 変更者
- 状態（仮 / 確定）

---

## 16. 通知

通知イベント:

- 初回アカウント案内
- パスワード再設定
- KPI入力依頼
- KPI期限超過
- 評価未作成
- 面談実施依頼
- 面談後に評価公開
- 再印刷必要
- 評価方式変更
- 一次評価者変更
- マッピング未設定エラー
- 公開取消

送信先:

- 原則は本人 + 上長
- 公開通知 / 公開取消通知は本人のみ
- マッピング未設定エラーはシステム管理者のみ

テンプレート:

- 件名・本文はシステム管理者が編集可能

以下を監査ログ対象とする:

- ログイン
- KPI作成・修正
- 評価作成・修正
- 印刷
- 役員修正
- 給与正本更新
- 通知書公開 / 取消 / 再公開

閲覧権限は役員のみとする。

## 19. 推奨技術スタック

### 19-1. 推奨構成

- フロントエンド: Next.js（App Router） + TypeScript
- UI: React + Tailwind CSS
- バックエンド（フォーム操作）: Next.js Server Actions（KPI入力など、監査ログ不要の操作）
- バックエンド（業務操作）: Next.js Route Handlers（評価確定・公開・昇給確定など、監査ログを伴う操作）
- DB: PostgreSQL
- ORM: Prisma
- バリデーション: Zod（Server Actions / Route Handlers の入力検証、フォームバリデーションの共通化）
- データフェッチ管理: TanStack Query（React Query）（一覧・詳細のデータフェッチとキャッシュ管理）
- 認証: Auth.js（Credentials Provider） + bcrypt（セッションタイムアウトは無効に明示設定）
- メール送信: Nodemailer（SMTPベース）
- バックグラウンドジョブ: pg-boss（PostgreSQLベースのJobQueue。KPI期限超過・評価未作成などのスケジュール通知に使用。外部Redisが不要でDocker構成に収まる）
- PDF生成: Puppeteer（headless Chrome）+ Recharts（SVGレンダリング）によるサーバーサイドPDF生成（年次通知書の3軸推移グラフに対応）
- Redmine連携: REST API連携 + CSV代替取込
- 監査ログ / 履歴: DBテーブルで保持
- インフラ: Dockerコンテナ構成

### 19-2. 推奨理由

- このシステムは業務画面、権限制御、通知、帳票、履歴管理が主であり、Next.js + PostgreSQL の構成が最もバランスがよい
- フロントとバックを同一リポジトリで持てるため、制度変更時の改修速度が高い
- Prismaにより、マスタ・履歴・監査ログなど関係の強いデータを型安全に扱いやすい
- PostgreSQLは監査ログ、履歴、配分計算、検索条件の多い業務システムと相性がよい
- Docker化しておけば、将来クラウド移行・オンプレ運用のどちらにも寄せやすい
- Zodによりサーバー・クライアント間でバリデーションスキーマを共有でき、TypeScriptの型安全と一貫性を保てる
- pg-bossはPostgreSQLを使い回すため、Redisなど追加インフラが不要。スケジュール通知をDocker構成内に収められる
- PuppeteerはDocker環境前提ならコンテナイメージの重量（+500MB程度）が許容範囲であり、SVGグラフを含むHTMLをそのままPDF化できるため帳票品質が高い

### 19-3. Server Actions と Route Handlers の使い分け方針

- Server Actions を使う操作: KPI入力・保存、コメント保存、プロフィール変更など、フォームから直接呼び出せるシンプルな操作
- Route Handlers を使う操作: 評価確定・公開・公開取消・昇給配分確定・給与正本更新など、監査ログ記録が必須の操作。IPアドレスや操作者情報を確実に取得するためRoute Handlersで統一する

### 19-4. 技術選定の基本方針

- 先に業務ルールとデータ構造を固定し、UIとDBスキーマをぶらさない
- 評価方式は固定仕様とし、方式追加はシステム改修前提で扱う
- 添付ファイル機能は初期スコープから外し、コメント中心で先に安定運用を実現する
- DB設計は `docs/db_style_guide 3.md` を準拠ガイドとして行う
  - PostgreSQL 前提の命名規則、テーブル種別プレフィックス、カラム命名、履歴・ログ設計方針を優先適用する
  - 本仕様書とガイドに差異がある場合は、本仕様書の業務要件を優先しつつ、命名・実装様式は可能な限りガイドへ合わせる
